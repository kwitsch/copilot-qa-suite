<task>
1. Locate the functions or methods affected by the CHANGED_LINES range
   in <context>.
2. Detect the test framework from build/config files in the project:
   - build.gradle, build.gradle.kts, pom.xml -> JUnit4/5, Kotest, TestNG, Spock
   - package.json (devDependencies) -> Jest, Vitest, Mocha, Playwright, node:test
   - pyproject.toml, setup.py, requirements*.txt -> pytest, unittest
   - go.mod -> Go builtin testing
   - Cargo.toml -> Rust builtin #[test] / rstest
   - composer.json -> PHPUnit, Pest
   - Gemfile -> RSpec, minitest
3. Determine the project's test-file convention from existing tests:
   - Kotlin/Java (Gradle/Maven): src/test/kotlin or src/test/java,
     class name + "Test"
   - Python pytest: tests/test_<n>.py or test_<n>.py next to module
   - TS/JS Jest/Vitest: <n>.test.ts / <n>.spec.ts next to module
     or under __tests__/
   - Go: <n>_test.go in the same package
   - Rust: #[cfg(test)] mod tests {} in the same file OR tests/<n>.rs
   - PHP: tests/<n>Test.php
   - Ruby: spec/<n>_spec.rb
4. Check whether tests already cover the changed logic.
5. If there is a coverage gap, propose concrete tests.

Skip (output NO_FINDINGS, no suggestions):
- Pure data classes (data class, @dataclass, structs without methods)
- Enums without behaviour
- Trivial getters/setters with no logic
- Private helpers whose code paths are already covered through public
  API
- Generated code, DTOs without logic
- UI components (React / Compose / SwiftUI) — integration / E2E scope
- Main / launcher functions, plain logger calls
- Simple constructor assignments
</task>

<no_comments_rule>
The generated test code MUST NOT contain ANY comments. No KDoc /
Javadoc / JSDoc / TSDoc / godoc / rustdoc blocks above test
functions, no inline // or # line comments inside test bodies, no
block comments. The ONLY exception is `package` / `import` lines and
test-framework annotations themselves (@Test, #[test], etc.) which
are not comments.

Reason: documentation is added afterwards by the doc hook when it
fires on the newly created test file. Including comments now would
produce duplicate (and likely worse) docs and waste the
additionalContext budget.

If you would normally explain WHY a test exists, encode the rationale
into the test method name instead. Prefer
`testReturnsZeroForEmptyCart` over `// returns 0 when cart is empty`.
</no_comments_rule>

--- END ---

Severity rules:
- CRITICAL: new function with relevant logic and no tests at all;
            existing test broken by signature change;
            security/correctness path with no test.
- WARNING:  change affects edge behaviour (boundary, error path,
            null/empty) but existing test only covers happy path.
- INFO:     additional edge-case tests would be useful but not
            critical.

auto_fix=YES only when ALL of the following hold:
  - Framework is unambiguously detected (not UNKNOWN).
  - Test file path is clearly derivable from project convention.
  - Proposed tests are isolated and deterministic.
  - No mocking of external systems (HTTP, DB, filesystem) needed,
    OR the project's existing test setup is obviously reusable.
  - No restructuring of existing tests required.

If no tests are needed (or none can be sensibly automated), output
EXACTLY this single line and nothing else:

NO_FINDINGS
