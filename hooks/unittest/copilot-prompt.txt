NO COMMENTS RULE (IMPORTANT)
============================
The generated test code MUST NOT contain ANY comments. No KDoc/Javadoc/
JSDoc/TSDoc/godoc/rustdoc blocks above test functions, no inline `//`
or `#` line comments inside the test bodies, no block comments. The
ONLY exception is the `package` / `import` lines and the test framework
annotations themselves (`@Test`, `#[test]`, etc.) which are not
comments.

Reason: documentation is added afterwards by the doc hook when it
fires on the newly created test file. Including comments now would
produce duplicate (and likely worse) docs and waste the 10 KB
additionalContext budget.

If you would normally explain WHY a test exists, encode the rationale
into the test method name instead. For example, prefer
`testReturnsZeroForEmptyCart` over a comment `// returns 0 when cart
is empty`.

SEVERITY RULES
==============
- CRITICAL: new function with relevant logic and no tests at all;
            existing test broken by signature change;
            security/correctness path with no test
- WARNING:  change affects edge behavior (boundary, error path,
            null/empty) but existing test only covers happy path
- INFO:     additional edge-case tests would be useful but not critical

AUTO_FIX RULES
==============
auto_fix=YES only when ALL of the following hold:
  - Framework is unambiguously detected (not UNKNOWN)
  - Test file path is clearly derivable from project convention
  - Proposed tests are isolated and deterministic
  - No mocking of external systems (HTTP, DB, filesystem) needed,
    OR the project's existing test setup is obviously reusable
  - No restructuring of existing tests required

auto_fix=NO when:
  - Framework=UNKNOWN
  - Integration or E2E test is needed
  - Complex mocking / test containers / fixtures
  - Existing tests would need to be restructured
  - Project convention is ambiguous
