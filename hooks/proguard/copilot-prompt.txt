TASK
====
1. Read the changed Gradle file completely.
2. For `build.gradle[.kts]`: extract every entry in `dependencies {}`
   blocks (implementation, api, kapt, ksp, annotationProcessor; ignore
   testImplementation and androidTestImplementation for the unit-context
   scope). Honor version-catalog references (libs.xxx).
3. For `libs.versions.toml`: extract all defined libraries and their
   groups.
4. Use `read`/`grep` to load the contents of all listed ProGuard files.
5. Check whether ProGuard/R8 minification is actually active:
   - `minifyEnabled true` (Groovy DSL) or `isMinifyEnabled = true`
     (Kotlin DSL) inside a `buildTypes { release {} }` block
   - If minification is not active anywhere: output NO_FINDINGS.

6. For each dependency, check:
   a) Is it known to require consumer-side ProGuard rules (i.e. keep
      rules NOT included in the library's own consumer-rules.pro)?
      Common cases:

      - `com.squareup.retrofit2:retrofit` -> -keep for generic service
        interfaces, @Query/@Path annotation retention, response-model
        classes
      - `com.google.code.gson:gson` -> -keepclassmembers for model
        classes with @SerializedName, generic TypeAdapters
      - `com.fasterxml.jackson.core:jackson-databind` -> -keep for POJOs
        with Jackson annotations, enum values
      - `org.jetbrains.kotlinx:kotlinx-serialization-json` ->
        -keepattributes, -keep for @Serializable classes and their
        $Companion / $serializer
      - `com.squareup.moshi:moshi` -> -keep for generated *JsonAdapter
      - `androidx.room:room-runtime` -> -keep for @Entity, @Dao (when
        reflection is in use)
      - `com.google.dagger:hilt-android` / Dagger -> usually handled by
        the plugin, but check -keep on @Inject constructors
      - `com.google.firebase:firebase-*` -> Firebase-specific keep rules
      - `io.reactivex.rxjava3:rxjava` -> sun.misc.Unsafe keep not needed,
        but -dontwarn for missing jsr305 annotations
      - `com.squareup.okhttp3:okhttp` -> -dontwarn for optional
        dependencies
      - `org.conscrypt:conscrypt-android` -> specific keep rules
      - `androidx.navigation:navigation-*` -> -keep for NavArgs classes
      - `com.airbnb.android:epoxy` -> generated model classes
      - `com.jakewharton.timber:timber` -> usually covered
      - `org.greenrobot:eventbus` -> -keepclassmembers for @Subscribe
      - `com.google.protobuf:protobuf-*` -> -keep for generated proto
        classes

   b) Is the rule present in any of the ProGuard files? If not:
      severity=CRITICAL (runtime crash in release build is likely).

   c) Are there rules in ProGuard files whose corresponding library is
      NO LONGER in the dependencies (e.g. leftovers after migration)?
      severity=WARNING with action=REMOVE.

   d) Contradictory / duplicate / weaker-than-needed rules
      (e.g. `-keep class x.**` when a more specific `-keep class x.y.Z`
      already exists): severity=INFO.

7. IGNORE:
   - Pure stylistic differences (order, blank lines).
   - Debug-only rules (in `buildTypes { debug {} }` without
     minification).
   - Test dependencies for ProGuard analysis (do not affect release
     build).

OUTPUT FORMAT
=============
One block per finding. No preamble, no explanation outside the blocks.

--- FINDING ---
category: <MISSING_RULE | OBSOLETE_RULE | REDUNDANT_RULE | MISCONFIG>
library: <Gradle coordinates, e.g. "com.squareup.retrofit2:retrofit" or "n/a">
target_file: <absolute path of the ProGuard file to write to>
action: <APPEND | REMOVE | REPLACE>
existing_rule: <on REMOVE/REPLACE: the exact block from the file. Otherwise "(none)">
rules_block:
-keep class com.example.Foo { *; }
-keepclassmembers class com.example.Bar { *; }
severity: <CRITICAL | WARNING | INFO>
reason: <one-line explanation of why this change is needed>
source: <URL of the official library ProGuard documentation OR "own analysis">
auto_fix: <YES | NO>
--- END ---

AUTO_FIX RULES
==============
auto_fix=YES only when ALL of the following hold:
  - library is one of the cataloged known libraries (or one with public,
    documented official rules)
  - rules_block is unambiguous and small (no mass-keeps like
    `-keep class **`)
  - target_file exists and is not a foreign module's consumer-rules.pro
  - For APPEND: rules_block is not already in the file
  - For REMOVE: existing_rule is unambiguously identifiable

auto_fix=NO when:
  - Library is unknown and undocumented
  - Project-specific model classes whose package is not clearly derivable
  - Broad -keep rules with wildcards
  - The rule concerns reflection code in the application itself

NULL RESULT
===========
If no problems were found (or minification is not active), output
EXACTLY the following line and nothing else:

NO_FINDINGS
